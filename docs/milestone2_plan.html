Milestone 2 Plan

3D Representation and Mesh Extraction — Tiny3D-Latent

Goal

Choose the 3D format the model will learn and prove it can become a mesh.

The pipeline established in this milestone:

occupancy grid (.npy)  ->  marching cubes  ->  cleanup  ->  mesh.obj  ->  turntable GIF

This milestone exists because modern 3D generation is not only about tensors. The output must eventually become an inspectable, loadable, and renderable 3D asset. Establishing that asset path before training any neural network prevents debugging representation bugs and model bugs simultaneously.

Step-by-Step Implementation

Step 1: Marching Cubes Extraction

File: tiny3dlatent/representation/marching_cubes.py

  • Accept a 32³ uint8 occupancy grid and an iso-value (default 0.5).
  • Pad the grid with a 1-voxel empty border before extraction. This prevents open holes when a shape touches the grid boundary.
  • Run skimage.measure.marching_cubes to obtain vertices, faces, and normals.
  • Return a trimesh.Trimesh.

Step 2: Basic Mesh Cleanup

File: tiny3dlatent/representation/cleanup.py

Raw marching cubes output is often broken. Apply automatic hygiene in this order:

  1. mesh.merge_vertices() — remove duplicate vertices.
  2. mesh.remove_degenerate_faces() — remove zero-area triangles.
  3. mesh.remove_unreferenced_vertices() — cleanup after face removal.
  4. mesh.fix_normals() — ensure consistent, outward-facing normals.

Return the cleaned mesh. This keeps previews trustworthy and game-engine imports clean.

Step 3: OBJ Export

File: tiny3dlatent/representation/export.py

  • Export cleaned trimesh.Trimesh to a .obj file.
  • trimesh.exchange.obj.export_obj handles vertices, faces, and normals automatically.
  • No material file (.mtl) is produced yet — geometry only.
Why OBJ now? OBJ is universally supported by game engines (Unity, Unreal, Godot), Blender, and online 3D viewers. It is the safest baseline for an inspectable asset.

Step 4: Mesh Statistics

File: tiny3dlatent/representation/stats.py

Compute and return the following dictionary for every exported mesh:

FieldTypeWhy it matters
vertex_countintComplexity baseline for generated shapes
face_countintRendering cost proxy
bounds_min / bounds_max[x, y, z]Verify shape fits inside normalized unit cube
volumefloatSanity check (positive for closed meshes)
surface_areafloatAnother sanity check
is_watertightboolCritical for 3D printing and some game uses
file_size_bytesintStorage cost of exported asset

Save the dictionary as mesh_stats.json next to each mesh.obj.

Step 5: Turntable GIF Preview

File: tiny3dlatent/representation/preview.py

Implementation strategy (matplotlib 3D + imageio, no display server required):

  1. Render the mesh using matplotlib.axes.Axes.plot_trisurf.
  2. Rotate the camera azimuth from 0° to 360° in N steps (default 36 frames = 10° per step).
  3. Capture each frame to an in-memory RGB buffer.
  4. Assemble all frames into a GIF using imageio.mimsave.

Benefits: guaranteed to work on macOS, no OpenGL context needed, easy to embed in reports.

Future upgrade: If pyrender installs cleanly later, the same preview.py interface can be swapped to a shaded, lit renderer without changing callers.

Step 6: CLI Entry Point

File: tiny3dlatent/experiments/mesh_extraction.py

Command:

python -m tiny3dlatent.experiments.mesh_extraction --config configs/mesh_extraction.json

Config (configs/mesh_extraction.json):

{
  "source_dataset": "data/procedural",
  "split": "val",
  "num_examples": 6,
  "resolution": 32,
  "iso_value": 0.5,
  "cleanup": true,
  "turntable_frames": 36,
  "random_seed": 0
}

Behavior:

  1. Read data/procedural/metadata.json.
  2. Filter to the chosen split (train or val).
  3. Pick num_examples records reproducibly using random_seed.
  4. For each shape:
    • Load the .npy occupancy grid.
    • Extract mesh via marching cubes.
    • Run cleanup (if cleanup: true).
    • Export mesh.obj.
    • Compute stats → mesh_stats.json.
    • Render turntable → turntable.gif.
  5. Write a run_summary.json aggregating all per-shape stats for easy comparison.

Deferred Work (Post-Milestone 2)

These items are intentionally out of scope for the initial implementation. They can be added later without changing the core pipeline:

  • Deferred TSDF representation: A tiny3dlatent/representation/tsdf.py module that converts occupancy grids to signed-distance grids using scipy.ndimage.distance_transform_edt, then runs marching cubes. This would allow a side-by-side comparison of blocky occupancy meshes vs. smoother TSDF meshes.
  • Deferred Shaded turntable: Swap the matplotlib renderer for a pyrender-based lit renderer if/when it installs cleanly.
  • Deferred Material export: Add a basic .mtl file so the OBJ carries a diffuse color based on the shape label (e.g., red sphere gets a red material). This is useful for Milestone 7 (Color and Simple Material Channels).

Design Decisions

Why Cleanup Belongs in Milestone 2

Milestone 2 is responsible for producing a trustworthy, inspectable asset. Raw marching cubes output often contains duplicate vertices and degenerate faces. Without cleanup:

  • Mesh statistics (vertex count, volume, watertightness) are unreliable.
  • Game engines may import with visible seams or missing faces.
  • You may waste time in Milestones 3–5 debugging "bad autoencoder outputs" that are actually export artifacts.

Basic cleanup is ~4 lines of trimesh calls. It is hygiene, not analysis. Milestone 8 will later use the same cleanup code but add diagnostic metrics (connected components, IoU, failure classification) on top.

Why Matplotlib Instead of PyRender for the Turntable

pyrender requires an OpenGL context and a display server, which is fragile on headless and macOS environments. Matplotlib 3D is slower and less photorealistic, but it is guaranteed to work everywhere. The turntable GIF is a proof-of-3D-inspection, not a portfolio beauty render. If higher visual quality is needed later, the renderer can be swapped behind the same function signature.

Why 6 Validation Examples

Validation split already exists from Milestone 1. Six shapes is enough to cover the six shape families (sphere, cube, rounded_box, cylinder, capsule, torus) and verify that every family extracts cleanly. It is small enough to run in seconds but large enough to catch shape-specific bugs (e.g., a torus with a thin ring might need a finer iso-value).
